- 修改代码前，先完整阅读相关文件，理解上下文再动手
- 不要猜测 API 签名或函数参数，先查阅实际代码或文档
- 每次修改后，主动说明做了什么、为什么这样做
- 如果任务不明确，先提问澄清，不要擅自假设意图

## Code Style

- TypeScript 严格模式，禁止使用 `any`，必须明确类型
- 优先使用现有的工具函数，不要重复造轮子
- 函数保持单一职责，超过 50 行考虑拆分
- 错误处理要显式，不要吞掉 catch 块
- 注释写"为什么"，而不是"做了什么"（代码本身说明做了什么）

## Git Workflow

- 每个功能/修复对应一个独立分支，命名格式：feat/xxx 或 fix/xxx
- commit message 使用约定式提交格式（Conventional Commits）
- 不要在主分支直接提交，通过 PR 合并
- rebase 前确认分支状态，使用 --force-with-lease 而非 --force
- PR 描述中说明：改了什么、为什么改、如何测试

## Testing

- 新增功能必须附带对应测试，不接受无测试的 PR
- 测试文件与源文件放在同一目录，命名为 *.test.ts
- 测试描述用业务语言写，而不是技术实现细节
- Mock 外部依赖，不要让单元测试依赖网络或数据库
- 修复 bug 前先写一个能复现问题的失败测试

## Agent Development Rules

- 工具函数（tool）必须有清晰的 description，这是模型理解意图的关键
- 每个 tool 只做一件事，复合操作拆成多个 tool 组合调用
- 状态变更操作（write/delete）需要在 description 中明确标注副作用
- Agent 的系统提示和用户提示分开管理，不要混写
- 长任务必须有 checkpoint，支持中断恢复，不要全量重跑
- 记忆写入前做去重检查，避免向量库中存入冗余内容
